# Body dimension build — ⚑ judgment calls awaiting Dylan

> Appended by each build pass of the Body dimension (spec: `planning/dimension-body-build.md`).
> One line per call: what was decided and why. Blessing or veto happens in review — nothing
> here silently reinterprets the spec.

⚑ [P1a] Hold sets store `reps: 0` (LiftingBlock keeps required `reps`, no discriminated union) — 0 is the honest rep count for a pure isometric hold, and volume math (weightKg × reps) then contributes nothing by construction.
⚑ [P1a] Hold sets count toward per-pattern set counts + a new `holdSecByPattern` seconds aggregate, ZERO volumeLoadKg — inventing a load-equivalent for holds would be fabricated volume (spec-carried ⚑, recorded here for review).
⚑ [P1a] revealLifting: a hold-only session says "N s held" and drops the volume-load segment; any rep work (even 0 kg bodyweight) keeps the "kg volume load" figure exactly as before — "0 kg volume load" on a pure-hold session would read as no work done.
⚑ [P1a] On a hold set an EMPTY weight field = strict bodyweight (stored 0 kg added load); the rep-set rule is unchanged (weight must still be typed, 0 = bodyweight) — loosening the rep-set rule too would silently change existing validation behavior.
⚑ [P1a] breathwork + pt land on modality 'mobility' (spec-carried ⚑) — identity-level distinctness is what matters; 'mobility' is the nearest coarse engine bucket and nothing downstream needs a dedicated value.
⚑ [P1a] New-activity details: breathwork icon 'wind' / tags ['mindfulness']; pt label "PT" / icon 'heart-pulse' / tags ['recovery'] — label + tags deliberately avoid rehab*/rx*/therapy wording (FDA wellness naming rules, pt-model.md).
⚑ [P1a] headlineActivities filters deprecated ids even from a stored preference list, not just the defaults — a stale saved headline row can't resurface martial-arts in a picker.
⚑ [P1b] breathwork.capture is written only when the capture UI set one AND rounds exist — an unset field stays absent (unknown provenance), never a defaulted 'manual'; provenance/display only, never a tier or weight.
⚑ [P1b] The breathwork duration exemption is narrow: practice surface + activity 'breathwork' + ≥1 filled round + an EMPTY duration field; a typed duration must still parse > 0 — typed garbage errors instead of being silently dropped.
⚑ [P1b] Out-of-range pain (not a 0–10 integer) or tightness (not 1–5) BLOCKS the save with a message rather than being clamped or dropped — never rewrite what the user typed; untouched drafts (no zone or no score) drop at build like empty gym sets.
⚑ [P1b] painAreas lives on SessionPayload for ANY surface (a knee can hurt on a run or under a bar, per pt-model), not just practice; a standalone flare-up is a subjective observation with metric 'pain' — value 0 is a recorded pain-free reading, distinct from absent.
⚑ [P1b] getUserProtocols returns [] when unset (zero protocols is a fact) — a deliberate divergence from the null-until-set convention of the bodyProfile/appSettings tenants, where null gates an empty-state UI.
⚑ [P1b] A protocol tick = subjective observation {metric:'protocolTick', value:1, tier 1, fidelity 1}; the toggle is keyed on (protocolId, exerciseId) within the device-local civil day, and untoggle HARD-DELETES the row (consistent with the Pass 6 destructive-edit contract), never writes a 0.
⚑ [P1b] romReading is its own ObservationKind mirroring weighIn (not a subjective metric) — a measured test value with testId/side/unit is instrument-shaped like a weigh-in; serialize/storage needed zero changes (payload JSON passthrough), proven by real-SQL tests.

⚑ [P3] Manually editing the exercise name after a pick clears the now-stale `exerciseId` (`setExerciseName`) — a free edit is a free edit, not a mislabeled library row; re-picking is one tap away via the suggestions list.
⚑ [P3] Entry mode (reps vs hold-seconds) is a UI-only per-exercise field on the draft (`ExerciseDraft.entryType`), never persisted directly — each SET still independently derives its own kind from whichever of `reps`/`holdSec` is filled at build time. ⚑ [P3] Ladder engine: a loadable step's `effectiveLeverage` is computed PER SET (added load can vary set to set within one occurrence), not once per occurrence; a missing trend-bodyweight reading on a loadable-step date degrades the leverage to the step's raw factor (added load scores as 0 that day) rather than inventing a bodyweight.
⚑ [P3] L-sit chain's ladder-position "achieved" value is ACCUMULATED seconds across the occurrence's sets (Antranik's accumulate-60s rule, ladders-notes.md); every other chain (including dip-line's cross-metric hold→reps seam) uses the single best set. `bestScore`/`volumeScore` always report best-set and leverage-weighted-volume regardless of chain, so the raw numbers are never hidden even when position uses the accumulated view.
⚑ [P3] `useLadderProgress` only lists chains with at least one logged occurrence (Training tap-in `/training-progress`) — a never-touched chain is simply absent from the list, never a fabricated 0%-progress row.
⚑ [P3] Template `exerciseId` (sessionTemplate.ts) is schema-only this round — no picker UI in `edit-template.tsx` — but threaded losslessly through the form's inverse/build round-trip so a future writer's value on an existing template isn't silently dropped by an unrelated edit+save.

⚑ [P4] e1RM/PR exercise grouping keys on `exerciseId` first, normalized name fallback — same precedence as the P3 ghost resolver, kept consistent deliberately rather than inventing a second grouping rule.

⚑ [P4] A session with two improving sets on the same lift/weight produces exactly ONE flag per (exercise, kind) — the best qualifying set, not one flag per set that individually beat history. Avoids noisy double-counting within a single session.
