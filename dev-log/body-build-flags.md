# Body dimension build — ⚑ judgment calls awaiting Dylan

> Appended by each build pass of the Body dimension (spec: `planning/dimension-body-build.md`).
> One line per call: what was decided and why. Blessing or veto happens in review — nothing
> here silently reinterprets the spec.

⚑ [P1a] Hold sets store `reps: 0` (LiftingBlock keeps required `reps`, no discriminated union) — 0 is the honest rep count for a pure isometric hold, and volume math (weightKg × reps) then contributes nothing by construction.
⚑ [P1a] Hold sets count toward per-pattern set counts + a new `holdSecByPattern` seconds aggregate, ZERO volumeLoadKg — inventing a load-equivalent for holds would be fabricated volume (spec-carried ⚑, recorded here for review).
⚑ [P1a] revealLifting: a hold-only session says "N s held" and drops the volume-load segment; any rep work (even 0 kg bodyweight) keeps the "kg volume load" figure exactly as before — "0 kg volume load" on a pure-hold session would read as no work done.
⚑ [P1a] On a hold set an EMPTY weight field = strict bodyweight (stored 0 kg added load); the rep-set rule is unchanged (weight must still be typed, 0 = bodyweight) — loosening the rep-set rule too would silently change existing validation behavior.
⚑ [P1a] breathwork + pt land on modality 'mobility' (spec-carried ⚑) — identity-level distinctness is what matters; 'mobility' is the nearest coarse engine bucket and nothing downstream needs a dedicated value.
⚑ [P1a] New-activity details: breathwork icon 'wind' / tags ['mindfulness']; pt label "PT" / icon 'heart-pulse' / tags ['recovery'] — label + tags deliberately avoid rehab*/rx*/therapy wording (FDA wellness naming rules, pt-model.md).
⚑ [P1a] headlineActivities filters deprecated ids even from a stored preference list, not just the defaults — a stale saved headline row can't resurface martial-arts in a picker.
⚑ [P1b] breathwork.capture is written only when the capture UI set one AND rounds exist — an unset field stays absent (unknown provenance), never a defaulted 'manual'; provenance/display only, never a tier or weight.
⚑ [P1b] The breathwork duration exemption is narrow: practice surface + activity 'breathwork' + ≥1 filled round + an EMPTY duration field; a typed duration must still parse > 0 — typed garbage errors instead of being silently dropped.
⚑ [P1b] Out-of-range pain (not a 0–10 integer) or tightness (not 1–5) BLOCKS the save with a message rather than being clamped or dropped — never rewrite what the user typed; untouched drafts (no zone or no score) drop at build like empty gym sets.
⚑ [P1b] painAreas lives on SessionPayload for ANY surface (a knee can hurt on a run or under a bar, per pt-model), not just practice; a standalone flare-up is a subjective observation with metric 'pain' — value 0 is a recorded pain-free reading, distinct from absent.
⚑ [P1b] getUserProtocols returns [] when unset (zero protocols is a fact) — a deliberate divergence from the null-until-set convention of the bodyProfile/appSettings tenants, where null gates an empty-state UI.
⚑ [P1b] A protocol tick = subjective observation {metric:'protocolTick', value:1, tier 1, fidelity 1}; the toggle is keyed on (protocolId, exerciseId) within the device-local civil day, and untoggle HARD-DELETES the row (consistent with the Pass 6 destructive-edit contract), never writes a 0.
⚑ [P1b] romReading is its own ObservationKind mirroring weighIn (not a subjective metric) — a measured test value with testId/side/unit is instrument-shaped like a weigh-in; serialize/storage needed zero changes (payload JSON passthrough), proven by real-SQL tests.
⚑ [P2] Hand-review split: 135 REVIEWED ids (auto-fill silently) / 738 unreviewed (prefill, visibly editable in the picker flow) — spec asked for ~100; the extra 35 are the family variants that came free while reviewing (bench/squat/deadlift variants), each still individually judged.
⚑ [P2] pickerScope hides 137 rows: all 123 stretching (which subsumes the 13 -SMR rows) + all 14 cardio — none are set-loggable; 'other'-equipment strength rows stay 'gym' per spec.
⚑ [P2] All four Free-Exercise-DB vendor-time additions live in CODE (exerciseLibrary.ts), the JSON is vendored verbatim — equipment patched ONLY on the 14 strength/plyo ids; the other 63 null-equipment rows are hidden stretching/cardio and stay null rather than fabricate a value nobody consumes.
⚑ [P2] body-only rows map to pickerScope 'both' (gym AND calisthenics pickers); the standalone 'calisthenics' value is reserved and unused today — first honest use is ladder-step library entries.
⚑ [P2] ladders-notes.md vendored ALONGSIDE the data at src/data/ (not planning/) — the leverage-factor rationale is dataset documentation, not a Dylan-blessed spec, and must move with the JSON it explains. loadable KEPT on exactly weighted dip + weighted pull-up, integrity-tested.
⚑ [P2] The three yoga upstream corrections are applied in LOADER code with UPSTREAM CORRECTION comments (JSON cannot carry comments; the vendored file stays verbatim-diffable against upstream). 48 SVGs snapshotted to assets/yoga-poses/ (~340 KB, author's personal cloudinary would otherwise be a runtime dependency); PNGs not vendored; the pose BROWSE surface stays deferred to the Gorge redesign.
⚑ [P2] entryType 'duration' seeded from force==='static' (10 non-hidden statics, e.g. Plank/Side_Bridge/Plate_Pinch; the other 94 statics are hidden stretching rows) plus the review pass; entryType ABSENT means reps-default — no value fabricated for the ambiguous rows.
⚑ [P3] Ladder-step movement patterns are hand-assigned per CHAIN (not derived), one line each in `exercisePicker.ts`'s `CHAIN_PATTERN` + `STEP_PATTERN_OVERRIDES` — levers/pulls → upper-pull, planche/handstand/pushup/dip → upper-push, L-sit/anti-extension → core, squat/hinge → quad-dom/hip-hinge with single-leg steps overridden to unilateral-leg. All chain steps ship `patternReviewed: true` since a human (the build agent, following the ladder dataset's own leverage-ordering logic) judged each one — treat as reviewed-but-worth-a-spot-check, not machine-derived.

⚑ [P3] Ghost resolver (`useExercisePatternMemory`) keys on `exerciseId` first, normalized name as fallback, and groups a session's matching sets together (not one overwrite per flat set) so "last session's sets" is the exercise's whole group in that session, in logged order.
⚑ [P3] A library/ladder pick (`pickExercise`) only fills `movementPattern`/`entryType` when the exercise doesn't already have one set — mirrors pattern-memory's existing "never overwrite a choice the user already made" rule; a pick always fills `name`+`exerciseId` though, since picking IS the deliberate choice for those two fields.
⚑ [P3] Manually editing the exercise name after a pick clears the now-stale `exerciseId` (`setExerciseName`) — a free edit is a free edit, not a mislabeled library row; re-picking is one tap away via the suggestions list.
⚑ [P3] Entry mode (reps vs hold-seconds) is a UI-only per-exercise field on the draft (`ExerciseDraft.entryType`), never persisted directly — each SET still independently derives its own kind from whichever of `reps`/`holdSec` is filled at build time. On edit-load it's inferred from whether any restored set actually has `holdSec` set, never fabricated.
⚑ [P3] Ladder engine: a loadable step's `effectiveLeverage` is computed PER SET (added load can vary set to set within one occurrence), not once per occurrence; a missing trend-bodyweight reading on a loadable-step date degrades the leverage to the step's raw factor (added load scores as 0 that day) rather than inventing a bodyweight.
⚑ [P3] L-sit chain's ladder-position "achieved" value is ACCUMULATED seconds across the occurrence's sets (Antranik's accumulate-60s rule, ladders-notes.md); every other chain (including dip-line's cross-metric hold→reps seam) uses the single best set. `bestScore`/`volumeScore` always report best-set and leverage-weighted-volume regardless of chain, so the raw numbers are never hidden even when position uses the accumulated view.
⚑ [P3] `useLadderProgress` only lists chains with at least one logged occurrence (Training tap-in `/training-progress`) — a never-touched chain is simply absent from the list, never a fabricated 0%-progress row.
⚑ [P3] Template `exerciseId` (sessionTemplate.ts) is schema-only this round — no picker UI in `edit-template.tsx` — but threaded losslessly through the form's inverse/build round-trip so a future writer's value on an existing template isn't silently dropped by an unrelated edit+save.
⚑ [P3 build note] Live-simulator interactive verification is BLOCKED by shared infrastructure, not a Body defect: all four dimension worktrees (Sky/Water/Earth/Body) + benchmarks share one bundle id (`com.dylan.healthcoachproject`) on the one booted "iPhone 17" simulator, so whichever worktree last ran a native build "owns" the installed binary underneath everyone else's Metro/JS. Body's own `app.json`/`package.json` correctly declare `expo-camera` (confirmed), but the CURRENTLY installed binary was built from a different worktree and lacks the ExpoCamera native module — `expo-router` eagerly imports every route (incl. the unrelated `log-food.tsx`) at boot, so the whole app red-screens before any Body screen can be reached, regardless of whose code is correct. Rebuilding natively from this worktree would fix it but would also clobber the shared binary out from under any other session mid-verification — out of scope to do unilaterally. P3 checkpoint verification instead relied on: full jest (584 tests incl. 13 new ladder-engine tests) green, `tsc --noEmit` clean, and manual trace of every new prop/data path end-to-end. Recommend Dylan either dedicate one simulator per worktree or do a final interactive pass after all four dimensions converge on main.
